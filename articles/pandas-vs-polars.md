## はじめに

Python データ処理の歴史において、Pandas は10年以上にわたり業界の標準として君臨してきた。しかし、データ規模が指数関数的に増大し、モダンマルチコアプロセッサの性能を最大限に引き出す必要性が高まる中で、Polarsという新たな選択肢が登場し、その地位を揺るがしている。

この変化は、単なるライブラリ間の優劣の問題ではなく、以下の根本的なパラダイムシフトを意味する：

- **命令型(Imperative)から宣言型(Declarative)へ**
- **シングルスレッド処理からマルチスレッド並列処理へ**
- **NumPyベースからApache Arrowベースへ**

この記事では、両ライブラリのアーキテクチャ、実行モデル、性能から将来的な展望まで、最新の動向やベンチマークを交えて詳しく解説していく。

## アーキテクチャの根幹：実装言語とメモリモデルの差異

Pandas と Polars の性能差の根源は、それぞれの基礎となる言語とメモリ管理の手法にある。

### 実装言語

| 比較項目 | Pandas | Polars |
|----------|--------|--------|
| 主な実装言語 | Python, C, Cython | Rust |
| バックエンド | NumPy | Apache Arrow (Native) |
| メモリレイアウト | 主に行ベース / ブロック管理 | 列指向 |
| スレッドモデル | 原則としてシングルスレッド (GIL に制限) | デフォルトでマルチスレッド (Work-stealing) |
| 欠損値処理 | センチネル方式 (NaN によるキャストが発生) | 妥当性ビットマップ方式 (Native null) |
| インデックス | 明示的な Index/MultiIndex | 整数位置による管理 (明示的 Index なし) |

:::message
**💡 専門用語の補足**
- **GIL (Global Interpreter Lock)**: Pythonの仕組み上の制約。一度に1つのスレッドしかPythonコードを実行できないようにするロックのことで、CPUコアが複数あってもマルチスレッドでの並列処理がしにくい大きな原因となる。
- **Work-stealing (ワークスティーリング)**: 並列処理の最適化アルゴリズムの一つ。早く自分のタスクを終えて手が空いたスレッドが、まだ忙しい他のスレッドの待ち行列からタスクを「盗んで（stealing）」代わりに実行する仕組み。これによりCPU全体が無駄なくフル活用される。
- **センチネル方式**: 欠損値（データが存在しない状態）を表すために、特定の値（`NaN` など）を「目印（センチネル）」として代入する方式。整数列に `NaN` が入ると全体が浮動小数点型に変換されてしまうなど、意図せぬ型変換を引き起こす原因となる。
- **妥当性ビットマップ方式**: データ本体とは別に、「その行のデータが存在するかどうか」を0と1のフラグ（ビットマップ）として独立して管理する方式。データ本体の型（整数など）を崩すことなく、綺麗かつ高速に欠損値を扱える。
:::

### メモリモデル

**Pandas のアプローチ:**
- NumPy 配列に基づくデータ表現
- 行ベースの処理や連続したメモリブロックの確保に優れる
- 文字列や欠損値（null）の混在する表形式データでは効率性に欠ける

**Polars のアプローチ:**
- Apache Arrow を最大限に利用する「列指向」フォーマット
- 同一列のデータがメモリ上で連続して配置
- 現代の CPU のキャッシュ効率を劇的に高め、SIMD 命令の活用を容易にする

### Apache Arrow のメリット

Arrow の採用は、単なる処理速度の向上にとどまらない：

1. **ゼロコピーデータ受け渡し**: Polars、Spark、DuckDB などの他のデータ処理エンジンとの間で、データのコピーやシリアライズを介さずにメモリ上のポインタを共有可能
2. **大規模データパイプライン**: ライブラリ間のデータ変換に伴うオーバーヘッドが事実上解消

### 欠損値処理の差異

**Pandas の問題点:**
- 整数列に欠損値が含まれる場合、浮動小数点数（float）にキャストして NaN を代入
- 型の一貫性を損ない、計算精度やメモリ使用量に悪影響

**Polars の解決策:**
- Arrow の仕様に基づき、各データ型に対して独立した「妥当性ビットマップ」を保持
- 元のデータ型を維持したまま欠損値を管理
- SQL セマンティクスに近い厳密かつ高速な処理が可能

## 実行モデルの変革：Eager から Lazy、そしてクエリ最適化へ

### 実行タイミングの違い

| 特徴 | Pandas | Polars |
|------|--------|--------|
| 実行モデル | Eager execution（即時実行） | Eager + Lazy execution（遅延実行） |
| 中間データ | 各操作で生成 | 最適化後に一括生成 |
| メモリ効率 | 低い（不要なコピー発生） | 高い（最適化により削減） |

:::message
**💡 実行モデルに関する専門用語の補足**
- **Eager execution (即時実行)**: 1行コードを書くごとに、すぐにその場で逐次計算を走らせる方式。結果がすぐに見えて試しやすい反面、処理全体を見渡した最適化はできない。
- **Lazy execution (遅延実行)**: すぐには計算を行わず、「どんな処理をしたいか」という計画（クエリプラン）だけを先に溜め込む方式。最後に結果を要求されたタイミングで、計画全体を見て最も効率の良い方法を考えてから一気に計算を行うため、非常に高速でメモリの無駄がない。
:::
**Pandas（Eager execution）:**
- コードの各行が記述された瞬間に計算が実行
- 中間結果としての DataFrame がメモリ上に生成
- **利点**: 対話的なデータ探索（EDA）で即時フィードバック
- **欠点**: メモリ消費が大きく、計算リソースの浪費が課題となるケースがある。

**Polars（Lazy execution）:**
- 一連の操作はすぐには実行されず、論理クエリプラン（DAG）として蓄積
- `.collect()` 呼び出し時にクエリ最適化エンジンが最適物理実行プランを生成
- **利点**: 宣言的に「何を取得したいか」を記述（SQL に近いアプローチ）

### クエリ最適化エンジンの主要な手法

Polars のクエリ最適化エンジンは、データベース管理システム（DBMS）で培われた高度な技術を DataFrame 操作に応用している。

#### 1. 述語プッシュダウン（Predicate Pushdown）

フィルタリング操作をデータソースの読み込み段階までさかのぼらせて適用：

```python
# 例：100GB の Parquet ファイルから特定の日付のみ抽出
df.filter(pl.col("date") == "2024-01-01").collect()
```

- Parquet のメタデータを利用して関連するブロックのみをスキャン
- I/O とメモリ使用量を劇的に削減

#### 2. 射影プッシュダウン（Projection Pushdown）

クエリの最終結果に必要な列のみを特定し、不要な列の読み込みを回避：

- 数百の列を持つデータセットで、実際に使用する 3 列のみを読み込み
- メモリフットプリントとディスクスキャン時間を最小化

#### 3. 演算の融合（Operation Fusion）

隣接する複数のフィルタリングや列演算を一つの処理ステップに統合：

- データのスキャン回数が減少
- CPU のキャッシュ効率が最大化

## 性能比較とリソース効率の検証

### ベンチマーク結果

Polars の性能的優位性は、多様なベンチマーク結果によって実証されている。特に、大規模なデータセットにおける結合（Join）や集計（Aggregation）操作において、その差は顕著である。

**TPC-H（PDS-H）ベンチマーク（2025 年）:**
- 環境：96 vCPU、192 GB RAM
- Pandas 2.2.3: 約 365.71 秒
- Polars ストリーミングモード: わずか 3.89 秒
- **速度差：約 94 倍**

:::message
**💡 専門用語の補足**
- **TPC-H ベンチマーク**: データベースやデータ処理エンジンの性能を評価するための「世界共通の体力テスト」のようなもの。実際のビジネスの現場で発生するような、極めて大容量のデータに対する複雑な集計や結合（Join）処理などの重い負荷を想定したテストとなっており、データ分析エンジンの真の実力が問われる指標となっている。
:::

### 実務的な性能比較（1,000 万行規模）

| 処理内容 | Pandas 実行時間 (秒) | Polars 実行時間 (秒) | 速度向上倍率 |
|----------|---------------------|---------------------|--------------|
| データ読み込み (CSV) | ~14.14 | ~1.25 | ~11.3x |
| フィルタリング | ~0.43 | ~0.002 | ~215x |
| 列の追加・演算 | ~0.015 | ~0.002 | ~7.5x |
| グループ化集計 (GroupBy) | ~9.21 | ~0.8 | ~11.5x |
| 文字列操作 | ~3.21 | ~0.03 | ~107x |

### メモリ効率の比較

| 指標 | Pandas | Polars |
|------|--------|--------|
| メモリ必要量 | データセットの 5〜10 倍 | データセットの 2〜4 倍 |
| 大規模データ処理 | メモリ不足でクラッシュ | ストリーミング API で対応 |
| 中間データ | 都度生成 | 最適化により削減 |

**Polars のストリーミング API:**
- 利用可能なメモリを超えるデータに対してチャンク単位で処理
- 必要に応じてディスクへ一時退避（Spill-to-disk）
- プロセスをクラッシュさせることなく処理を続行可能

## Pandas 3.0：伝統的ライブラリの技術的刷新

Polars などの新世代エンジンの台頭を受け、Pandas もまた、そのアーキテクチャの抜本的な改善を余儀なくされた。2026 年 1 月にリリースされた Pandas 3.0 は、長年の課題であったメモリ管理とデータ型の問題を解決するための「Arrow 革命」を完成させている。

### Copy-on-Write (CoW) のデフォルト化

**従来の問題点:**
- スライス操作の結果が「ビュー」か「コピー」かが状況により異なる
- 予期せぬデータの書き換えや SettingWithCopyWarning の混乱

**CoW の解決策:**
- 実際の内容が書き換えられるまでメモリ上の同じデータブロックを共有
- 変更が必要になった瞬間にのみコピーを作成
- 不要なメモリコピーが回避され、大規模データフレームに対する読み取り専用操作のパフォーマンスが大幅に向上

**効果:**
- 特定の ETL パイプラインでコードを変更せずに 38% の高速化

### Arrow バックエンドへの全面移行

**文字列データの改善:**
- NumPy の object 型 → Arrow バックエンドの専用 str 型
- 処理速度：5 倍〜10 倍向上
- メモリ使用量：劇的に削減

**新しい式構文:**
```python
# pd.col() の導入（Polars の pl.col() に酷似）
df.assign(new_col=pl.col("existing_col") * 2)
```

### バージョン別の技術的変化

| バージョン | 主な技術的変化 | メモリ管理の影響 |
|------------|---------------|------------------|
| Pandas 1.x | NumPy 主体、Sentinel NaN | コピーが頻繁に発生、メモリ消費大 |
| Pandas 2.0 | PyArrow バックエンドの試験的導入、CoW のオプション化 | Arrow 型によるメモリ効率改善の開始 |
| Pandas 3.0 | CoW の強制、Arrow 文字列のデフォルト化、pd.col 導入 | 参照共有による効率化、文字列操作の高速化 |

## GPU アクセラレーションと分散処理のフロンティア

データ処理の規模が単一マシンの CPU 限界を超えるケースに対応するため、Polars は GPU 活用とクラウドスケールへの対応を進めている。

### NVIDIA RAPIDS cuDF による GPU 駆動

**機能:**
- Lazy API を介した GPU アクセラレーション
- `.collect(engine="gpu")` で自動的に GPU へオフロード

:::message
**💡 専門用語の補足**
- **GPU アクセラレーション**: 本来は画像処理などに使われる「グラフィックボード（GPU）」の圧倒的な並列計算能力を、大量のデータ処理に転用して高速化させる技術。
- **Lazy API を介した〜とは**: 「遅延実行（Lazy execution）」の仕組みを活かし、「どのような計算をしたいか」という計画（クエリプラン）を、GPUが一番効率よく実行できる手順へと自動で翻訳してくれる機能のこと。ユーザーは難しいGPU用のプログラムを書かなくても、いつものPolarsのコードの末尾を `.collect(engine="gpu")` にするだけで数倍〜数十倍の計算処理スピードの恩恵を受けられる。
:::
**性能:**
- 大規模な結合（Join）や複雑なグループ集計で威力を発揮
- NVIDIA H100（2022年に発表された、生成AI時代を代表する超高性能なデータセンター向けハイエンドGPU）環境で最大 13 倍の速度向上

**Unified Virtual Memory (UVM):**
- ホストメモリ（RAM）と VRAM を一つのアドレス空間として統合管理
- 例：141GB の VRAM を搭載した GPU で 300GB のデータセットを処理可能
- データのスワップに伴う性能低下は約 1.2 倍〜1.3 倍

### 分散処理と Polars Cloud

**特徴:**
- PySpark が提供するような分散処理の力を、Polars の直感的かつ高速な API で実現
- 「対角スケーリング」：垂直スケーリングと水平スケーリングを最適に組み合わせ
- 従来の Spark ベースのインフラに比べて複雑性とコストを大幅に削減

## エコシステムの相互運用性：Narwhals の役割

Pandas と Polars という強力な二つの選択肢が存在することは、ライブラリ開発者にとっては「どちらの DataFrame をサポートすべきか」というジレンマを生む。この問題を解決するために登場したのが**Narwhals**（ナルワルズ）である。

> **語源**: イッカク（Narwhal）に由来

### Narwhals の機能

- **Zero-dependency 互換レイヤー**: 特定の DataFrame ライブラリに依存しない
- **対応ライブラリ（Full API）**: Pandas, Polars, cuDF, Modin, PyArrow
- **対応ライブラリ（Lazy-only）**: Dask, DuckDB, Ibis, PySpark, SQLFrame, Daft
- **メリット**: データ変換なしで各ライブラリのネイティブな計算エンジンを呼び出し可能

### 実装上の利点

```python
# Narwhals を使用したライブラリ開発
import narwhals as nw

@nw.narwhalify
def process_data(df):
    return df.group_by("category").agg(nw.col("value").sum())
```

**採用事例:**
- Altair, Plotly といった可視化ライブラリ
- Bokeh, Panel, Marimo などの対話的フレームワーク
- Scikit-lego, Fairlearn などの機械学習ライブラリ
- Pandas から Polars への移行を進めるユーザーに対しても、追加の変換コストなしでグラフ描画機能を提供

### Narwhals と Streamlit + Plotly ユーザーへの推奨

**結論：Pandas のままで問題ない。**

| シナリオ | 推奨 | 理由 |
|---------|------|------|
| **小〜中規模データ** (<100 万行) | ✅ Pandas で OK | Pandas 3.0 で十分高速 |
| **Streamlit での EDA 用途** | ✅ Pandas で OK | 対話的な分析には Eager モデルが適している |
| **Plotly での可視化** | ✅ そのまま使用可能 | Narwhals 経由で Pandas DataFrame を直接サポート |

**Narwhals が解決する課題:**

Narwhals は**ライブラリ開発者向け**の互換レイヤーであり、エンドユーザーが直接使用するものではない。

```python
# Plotly 開発者の視点（Narwhals を使用）
import narwhals as nw

# Narwhals を使えば、Pandas でも Polars でも同じコードで動作
def create_plot(df):  # df は Pandas でも Polars でも OK
    nw_df = nw.from_native(df)
    # Plotly は Narwhals 経由で両方をサポート
```

**エンドユーザー（あなた）へのメリット:**
- ❌ Narwhals を直接使う必要はない
- ✅ Plotly が Narwhals を採用していることで、**将来 Polars に移行しても Plotly グラフがそのまま動作する**という安心感

## 実務領域における応用：ECサイトのログ解析とMLパイプライン

Polars のアーキテクチャは、データ処理の負荷が極めて高い巨大ECサイトの機械学習（ML）パイプラインにおいても革新をもたらしている。

### 大規模行動ログの高速な特徴量エンジニアリング

**背景:**
- ECサイトでは、ユーザーのクリック、カート追加、購入といった無数の行動ログ（クリックストリーム）が日々蓄積される
- 推薦システム（レコメンドモデル）の学習用特徴量を作る際、過去N日間のアイテム別クリック数やユーザーセッションごとの滞在時間などを計算する「ウィンドウ集計」が頻出する
- Pandasでは億単位のログに対する複数キーの集計処理が遅く、メモリ不足（OOM）でパイプラインが停止する課題が常態化しやすい

**Polars の効果:**
- 先進的なクエリ最適化とマルチスレッド処理により、大規模なグループ化やウィンドウ処理が爆速化
- `Expr`（式）を用いた記述により、SQLのような直感的かつ複雑な特徴量生成パイプラインをシンプルに実装可能

| 処理タイプ | 速度向上（Pandas比） |
|-------------|-------------|
| セッション別ユーザー行動集計 | 約 10 倍 |
| 時系列の移動平均・ウィンドウ計算 | 約 15 倍 |
| ユーザー情報とアイテムマスタの巨大Join | 約 20 倍 |

- **メモリ消費量の大幅削減（Lazy APIの効果）**: 中間データを都度生成せず、不要なカラムの読み込みもスキップされるため、巨大ログでもリソース効率が極めて高い。

### 推薦システム（バッチ推論）パイプラインの効率化

**適用フェーズ:**
- レコメンドエンジンのバッチ推論前に行われる、ユーザー特徴量・アイテム特徴量の結合（前処理）
- 推論後に出力された候補に対して、商品の在庫状況（Sold Outなど）やビジネスロジックを反映させて絞り込むランキングフェーズ（後処理）

**効果:**
- パイプライン全体のボトルネックになりがちな前処理・後処理の時間を短縮
- 定期実行バッチジョブ（例: 毎時更新されるおすすめ商品リスト）の完了時間を大幅に圧縮し、より「鮮度の高い」推薦が可能になる
- 処理リソース（CPU/メモリ）を節約できるため、大規模ワークロードにおいてクラウドコンピューティング費用の継続的な削減に直結する

### 実務ケーススタディ：Snowflake + Streamlit + Plotly 環境

クラウドデータウェアハウス環境での典型的な構成における選択基準：

| 構成要素 | 推奨 | 理由 |

1. まずは Pandas 3.0 を使用（Snowflake から `pd.read_sql()` で取得）
2. 性能ボトルネックが発生したら、その部分だけ Polars を検討
3. Plotly は Narwhals 対応済みなので、どちらでも動作する

## 移行の意思決定と戦略：いつどちらを選択すべきか

Pandas と Polars の比較検証から導き出される結論は、一方が他方を完全に駆逐するというものではなく、**ユースケースに応じた「適材適所」**の判断が求められる。

### Pandas を維持すべきシナリオ

| 状況 | 理由 |
|------|------|
| **小〜中規模のデータセット (<100 万行)** | データが数 GB 程度でメモリに余裕がある場合、Pandas の成熟したエコシステムと使い慣れた文法のメリットが性能向上を上回る |
| **対話的な探索的データ解析 (EDA)** | ステップバイステップでデータを確認し、グラフを書きながら分析を進める場合、即時実行モデルが快適 |
| **高度な時系列解析や特化型ライブラリへの依存** | Geopandas や特定の金融・統計解析パッケージなど、Pandas のインデックスやオブジェクト構造に深く依存しているツールを使用する場合 |

### Polars への移行を優先すべきシナリオ

| 状況 | 理由 |
|------|------|
| **大規模データの ETL 処理** | 数千万行から数億行のデータを処理し、SQL のような集計や結合が中心となるデータエンジニアリングのワークロード |
| **クラウドコストの最適化** | 処理時間とメモリ使用量がクラウド利用料に直結する本番環境のパイプライン |
| **メモリ制限のある環境** | コンテナ環境やローカルラップトップで、限られたリソースを最大限に活用して巨大な CSV や Parquet を処理する必要がある場合 |
| **Parquet/Arrow ベースのデータレイク** | データが既に列指向形式で保存されており、述語プッシュダウンの恩恵を最大限に受けられる場合 |

### 移行に必要な思考の転換

**命令型 → 宣言型:**

```python
# ❌ Pandas 風（命令型）
for idx, row in df.iterrows():
    df.loc[idx, 'new_col'] = row['col1'] + row['col2']

# ✅ Polars 風（宣言型）
df = df.with_columns(
    pl.col("col1") + pl.col("col2").alias("new_col")
)
```

- 行ごとにループを回したり、値を直接書き換えたりする手続き的な記述を捨てる
- `pl.col()` を用いた式（Expression）による変換パイプラインを構築することが真価を引き出す鍵

## 結論：Pandas 3.0 と Polars の最新比較（2026 年）

最新のリリース（2026 年 1 月）である Pandas 3.0 と Polars を比較すると、**依然として大規模なデータ処理や複雑な集計においては Polars が高速**という結論が一般的である。

ただし、Pandas 3.0 では根本的なアーキテクチャの刷新が行われたため、以前のバージョンほどの圧倒的な差は縮まりつつある。

### パフォーマンスの直接比較

**Polars の優位性:**
- Rust 製の並列実行エンジンを持ち、デフォルトで全 CPU コアを活用
- 大規模なデータセット（1,000 万行以上）での GroupBy や Join 操作では、Pandas 3.0 に対しても**5 倍〜20 倍以上の速度差**

**Pandas 3.0 の改善:**
- Copy-on-Write (CoW) のデフォルト化で不要なメモリコピーを排除
- 特定の ETL パイプラインで 38% の高速化
- 文字列操作は PyArrow バックエンド採用により 5 倍〜10 倍高速化

### 処理速度が逆転・接近するケース

| ケース | 勝者 | 理由 |
|--------|------|------|
| **小規模データ（10 万〜100 万行以下）** | Pandas 3.0 または互角 | Polars の並列処理オーバーヘッド |
| **Parquet 形式の読み書き** | 互角 | 両者とも PyArrow をバックエンド使用 |
| **Python カスタム関数（apply）** | 互角 | Python の GIL がボトルネック |

### メモリ効率の違い

| 特徴 | Pandas 3.0 | Polars |
|------|-----------|--------|
| 評価モデル | Eager execution（即時実行） | Lazy Evaluation（遅延評価） |
| 最適化 | CoW による改善 | 射影プッシュダウンなど |
| 中間データ | 生成されやすい | 最適化により削減 |

### 最終結論：どちらが「早い」か

**開発の早さ（生産性）:**
- すでに Pandas に慣れているチーム
- 小規模なデータ分析（EDA）
- 既存の機械学習ライブラリ（Scikit-learn 等）との連携が主
- → **Pandas**の方がトータルの作業時間は短くなる可能性

**実行の早さ（計算速度）:**
- 数百万行を超えるデータの加工
- 繰り返し実行される ETL パイプライン
- メモリ制限の厳しい環境での処理
- → **Polars**が圧倒的に有利

**目安（1M Rule）:**
- データ量が**100 万行を超えるかどうかが**、Polars への移行を検討する一つの目安

## 今後の展望：ハイブリッドなデータスタックへ

Pandas と Polars の対立軸は、Python データサイエンスの進化そのものを象徴している。

**Pandas の近代化:**
- NumPy ベースの遺産を引き継ぎつつ、Apache Arrow という新たな基盤を取り入れる
- 既存の広大なエコシステムとの互換性を保ちながら進化

**Polars の新基準:**
- Rust による究極の並列性能
- Lazy 実行による自動最適化
- GPU やクラウドへのシームレスな拡張性

**重要なアプローチ:**
単一のツールに固執するのではなく、**Apache Arrow という共通言語を通じて、目的別のハイブリッドなアプローチをとること**である：

- **前処理や大規模変換**: Polars
- **機械学習や詳細な統計解析**: Pandas

Narwhals のような抽象化レイヤーの成熟も手伝い、開発者は背後のエンジンの違いを意識することなく、プロジェクトの要件に最適な計算リソースを選択できる時代へと突入している。

データエンジニアおよびデータサイエンティストは、これらのアーキテクチャの違いを深く理解し、以下の要素に応じて柔軟にツールを使い分けることが求められている：

1. **データ規模**
2. **チームのスキルセット**
3. **インフラストラクチャの制約**

```
Step 1: 現状の Pandas コードを維持
        ↓
Step 2: ボトルネック箇所をプロファイリング
        ↓
Step 3: 重い処理のみ Polars で実装
        ↓
Step 4: Narwhals で抽象化（オプション）
```

このアプローチにより、リスクを最小限に抑えつつ、性能改善のメリットを享受できる。
